### 0.4 詳細設計決定（E 系, 2026-06-03 確定）

| # | 項目 | 決定 |
|---|------|------|
| E1 | FFI スレッドモデル | **プル型**: Rust コアが capture スレッド + リングバッファを所有し、バインディングは poll で取得。RT スレッドを GIL から切り離し音飛びを防ぐ（プッシュ型コールバックは採らない） |
| E2 | デバイス/プロセス識別 | **安定 ID 主体 + 既定デバイス追従モード**。プロセスは **PID + 名前フォールバック**（PID 揮発対策） |
| E3 | エラー復帰 | **自動再オープン（指数バックオフ）+ ウォッチドッグ（無音ストリーム死検知）+ 復帰イベント通知**。デバイス切断 / 既定切替 / スリープ復帰に対応 |
| E4 | 命名・リポ構成 | コア = **`flexaudio`（Rust）**、Python バインディング = **`pyflexaudio`**。既存 `process-audio-capture` リポは Rust コアに**吸収**（WASAPI 部を移植） |
| E5 | チャンク粒度 | **固定時間 20ms デフォルト + 設定可** |
| E6 | ギャップ時挙動 | **無音埋めで時間軸維持 + chunk 不連続フラグ + イベント通知** |
| E7 | バックプレッシャ | **DROP_OLDEST 継続 + ドロップ数メトリクス化 + ChunkDropped イベント** |
| E8 | ダウンミックス | **ITU-R BS.775 標準係数**（5.1→2ch 等）+ ソフトクランプ [-1,1] |
| E9 | pause/resume・ホットスワップ | **pause/resume 継続、ホットスワップは後フェーズ** |
| E10 | VAD アドオン接続 | アドオンが内部で **48k ステレオ → 16k mono に downmix/resample** して Silero 推論。**モデルはパッケージ同梱（DL しない ← オフライン前提）**。Silero ONNX は約 1.8MB と極小で同梱コスト無視可 |
| E11 | 権限フロー | **同期チェック + 非同期要求、拒否は Event 通知**。Info.plist/manifest 要件を各 OS で明文化 |
| E12 | テスト戦略 | **合成ソース + FFI 境界モック**でハードウェアなし CI。実機テストは `hardware` マーカーで分離 |
| E13 | 観測性 | **xrun/drop/レイテンシのメトリクス API + Rust→Python ログブリッジ**。**ローカルのみ（テレメトリ・遠隔送信なし ← オフライン前提）** |

### 0.6 OS 別実装の確定知見（2026-06-15・出典付き調査 Opus×3 体）

本文 §5/§6 の Rust 前提への全面改訂の元データ。各マスの採用技術・自プロセス除外手段・PTS 出所・最重要地雷を確定。

**Windows**（`windows-rs` 0.62 / `cpal`）
- マイク = cpal(WASAPI 共有)。システム/プロセス = windows-rs で WASAPI 自力。プロセス別 = Process Loopback（`ActivateAudioInterfaceAsync` + `AUDIOCLIENT_ACTIVATION_PARAMS` + `VIRTUAL_AUDIO_DEVICE_PROCESS_LOOPBACK`）。既存 C++ `LoopbackCapture.cpp` をほぼ 1:1 移植可
- **自プロセス除外** = Process Loopback の EXCLUDE(self)。**これにより「システム出力」も Process Loopback EXCLUDE(self) で実装でき、System/Process を同一 API パスに統一可能**
- PTS = `GetBuffer` の QPCPosition(100ns)。DevicePosition でギャップ検知。COM は capture スレッドで `CoInitializeEx(MTA)`、借用スレッドでは初期化しない
- 最重要地雷: ①Process Loopback = **Build 20348(Win11) 必須**【C5 で対応済】 ②classic loopback は無音時にデータ停止→silent-keepalive 必要（Process Loopback EXCLUDE 経由なら回避を検討） ③process loopback で `GetMixFormat` が `E_NOTIMPL`→`WAVEFORMATEX` 手動供給 ④ARM64 で loopback が 0 を返す事例報告→要実機検証
- 参考: `wasapi` crate(MIT) が process loopback 実装済→最良の参照実装

**macOS**（`objc2-core-audio` 0.3.2 / `cpal`）
- マイク = cpal(CoreAudio、f32 ネイティブ)。システム/プロセス = Core Audio Process Taps（`AudioHardwareCreateProcessTap` + `CATapDescription` + プライベート aggregate device + IOProc）
- **生 FFI 不要**: `objc2-core-audio` 0.3.2 が Tap API をフル網羅。`CATapDescription` の initializer 切替で system 全体/特定プロセス/自除外を作り分け
- 自プロセス除外 = `init(stereoGlobalTapButExcludeProcesses:[自 PID→AudioObjectID])`
- **権限 = `kTCCServiceAudioCapture`（画面収録権限ではない）**。Info.plist = `NSAudioCaptureUsageDescription`。公開 API で権限照会不可（拒否時は無音が返る前提で UX）
- PTS = IOProc の `inInputTime`（mHostTime/mSampleTime）。**信頼性は要実機検証**（参照実装が未使用）
- 後始末順序を厳守（Stop→DestroyIOProc→DestroyAggregate→DestroyTap）、aggregate は private 必須（クラッシュ時のデバイス一覧汚染防止）

**Linux**（`pipewire` 0.10 / `cpal`）
- マイク = cpal(ALSA host 経由、pipewire-alsa ブリッジ)。**`BufferSize::Fixed(480)` 必須**（Default は巨大バッファ地雷 #446）
- システム出力 = pipewire で既定 sink の monitor を `stream.capture.sink` + `target.object` で取得（**3 マス中最易**）
- プロセス別 = pipewire レジストリで対象出力ノード追跡 + port 単位リンク（`create_object("link-factory")`）。**fan-out 直リンク方式**（アプリは実スピーカーに鳴ったまま録れる）。マッチは `application.id`→`process.binary`→`application.name`（PID 不可）
- **自プロセス除外 = OS プリミティブ無し**→アーキで（自分専用 private null sink に自再生を流し、既定 sink monitor を録る）
- PTS = `pw_buffer.time`（safe ラッパー不在→薄い unsafe FFI 1 枚必要）。ギャップ = 差分法 + `SPA_META_HEADER` DISCONT
- 権限 = 非サンドボックスは不要（xdg-desktop-portal は将来 Flatpak/Snap 時のみ、今回 defer）

**横断（全 OS 共通）**
- **cpal はリサンプル/リミックスをしない** → 48k/stereo/f32 正規化はコアの責務（`rubato` + 自前 mix）。mono→stereo 複製、5.1→2ch ダウンミックス
- **PTS クロック非対称**: マイク(cpal)=相対/不透明タイムスタンプのみ、loopback 系=デバイスクロック PTS → コアが共通の単調クロックへ正規化、クロスストリーム厳密同期はベストエフォート
- ライセンス: 全依存 permissive。唯一 `audio_thread_priority` が **MPL-2.0**（回避可、cpal の `realtime-dbus` で代替検討）

### 0.7 コア/FFI アーキテクチャの確定知見（2026-06-15・Opus 調査）

**C ABI**（cbindgen 生成 `flexaudio.h`）
- opaque ハンドル `FlexCtx`(1プロセス1個) / `FlexStream`(1ソース=2ch)。両録りは FlexStream 2 本
- 全関数 `int32` エラーコード返し + out パラメータ。**コールバックは一切公開しない**（プル型の核心）
- 全 FFI 関数を `catch_unwind` で包む（panic の ABI 越境は UB）。thread_local に last-error
- `flexaudio_abi_version()` 必須先頭呼び出し。struct は末尾追記のみ（前方互換）
- 主要関数: ctx_create/destroy・enumerate・**permission_status/request（一級機能）**・stream_open/start/stop/close・**poll_chunk**（ノンブロッキング既定、timeout_ms>0 で消費側のみ待機）・poll_event
- `FlexChunk`: data(interleaved f32・コア所有・次poll/stopまで有効)・frames(960@20ms)・channels=2・pts_ns(正規化単調)・seq・flags(DISCONT/RECOVERED/SILENCE)・dropped_before

**リングバッファ（2 段構成）**
- RT 段: cpal/IOProc/PW コールバック → **rtrb 0.3.4**（SPSC・アロケート無し）に push のみ
- 加工段(取り込みスレッド): rtrb pop → **rubato 3.0** で 48k SRC → 2ch mix → 20ms(960frame) 切り出し → **ringbuf 0.4** チャンクリングへ
- DROP_OLDEST はチャンクリングで実施。捨てた数を dropped_before に反映、閾値超で ChunkDropped イベント
- 20ms 境界は **SRC 後**に 960frame で切る（rubato が端数を内部保持）

**Python バインディング**
- **PyO3 0.29 + maturin 1.13 + rust-numpy 0.29、abi3-py39 で単一 wheel**（ctypes 案は不採用）
- `pyflexaudio` は C ABI 経由ではなく **`flexaudio` コアを直接リンクする薄い PyO3 crate**（他言語は C ABI 経由）
- numpy: 既定は recv_into でコピー(7.68KB/chunk と安価)、ゼロコピー view は readonly オプション
- GIL: poll の待機のみ `allow_threads` で解放。RT/取り込みスレッドは GIL に一切触れない（構造的保証）

**スレッド/復帰**
- ストリームごと: RT コールバック → 取り込み+加工スレッド(1本) → poll。ctx 単位でウォッチドッグ 1 本(~250ms tick)
- `audio_thread_priority` は RT 段の cpal スレッドのみ昇格、加工スレッドは通常優先度
- 無音死検知(最終到着時刻 atomic・2s 閾値) + 自動再オープン(250ms→5s 指数バックオフ+ジッタ) + RECOVERED/DISCONT フラグ
- クロック正規化: device PTS と単調クロックのオフセットを open 時に取得。マイクは到着時刻近似。クロスストリーム同期はベストエフォート

**VAD アドオン**
- **`ort` 2.0(pyke) を Rust 側に持つ別 crate `flexaudio-vad`**。Silero ONNX(~1.8MB・MIT) を `include_bytes!` 同梱
- ⚠ **`ort` の `download-binaries` はオフライン契約違反** → `load-dynamic` + ベンダ済み ONNX Runtime(MIT) 同梱で回避。CI を完全オフライン化
- ONNX Runtime 本体も MIT

**cargo workspace 構成**
```
flexaudio/ (workspace, 非公開, MIT予定)
├ crates/ flexaudio-core(OS非依存: ring/SRC/mix/状態機械/クロック正規化/イベント)
│         flexaudio-os-{windows,macos,linux}(cfg 分離)
│         flexaudio-mic(cpal) / flexaudio(facade) / flexaudio-ffi(repr(C)+cbindgen→.h)
│         flexaudio-vad(ort+Silero, optional feature)
└ bindings/pyflexaudio (PyO3 + maturin, flexaudio 直接依存)
```
- OS module は `cfg` 分離で「全 OS が全 OS のコードをビルドしない」（依存も当該 OS でのみ引く）
- `process-audio-capture`(C++) は vendoring せずロジックを Rust 移植、`THIRD_PARTY_NOTICES` にクレジット
- 配布: maturin abi3 wheel を OS×アーキ別。ORT 共有 lib + Silero を同梱で完全オフライン

**ライセンス監査**
- 配布物に伝播する copyleft なし。要注意 2 件: **audio_thread_priority(MPL-2.0・cpal 経由で不可避→未改変厳守+NOTICES)** と **cbindgen(MPL・ビルド時のみ→配布物に入らない)**。`cargo-deny` で CI ゲート
- cpal は Apache-2.0 単独(MIT デュアルでない)が寛容で問題なし

### 1.3 設計哲学（最重要）

```
無駄な接続点を削減する。
依存採用は「自前実装の難易度」を必ず天秤にかけ、
正当化できないラッパーは外す。
```

**判断基準（採用 vs 自前）**:

| 観点 | 採用へ傾く | 自前実装へ傾く |
|------|-----------|--------------|
| ライブラリ規模 | 大規模（numpy 等） | 小規模（数百行で済む） |
| 抽象段数 | 1 段ラッパー | 2 段以上のラッパー |
| 使う機能の割合 | フル機能が必要 | 一部の API のみ |
| メンテナンス | 活発 | 停滞・1人メンテ |
| OS 固有制約への対応 | 既に解決済み | 自分の方が制御しやすい |
| 工数 | 高い | 数日〜2 週間以内 |

### 1.4 設計不変条件（破ってはいけないルール）

これらは「判断」ではなく、破ると即誤動作・クラッシュを招く**前提条件**。詳細な根拠は § 12 横断リスクを参照。

1. **リアルタイムコールバック内はノンブロッキング**: `put_nowait` + `DROP_OLDEST` のみ。重い処理は Pipeline Thread 側へ（PortAudio/WASAPI クラッシュ防止）
2. **オーディオデバイスは必ず共有モードで開く**: 排他モード（exclusive mode）は使わない。他アプリ・他 Session との同時録音を可能にし、締め出しを防ぐ
3. **自プロセス音声を全 OS で除外**: システム音声/プロセス別キャプチャ時、自プロセスの再生音を除外（フィードバックループ防止）。macOS は `setExcludesCurrentProcessAudio_` 済み、Win/Linux も同等処理を統一実装
4. **アーキ依存バイナリは対象アーキでビルド済みを保証**: ネイティブ依存（特に ProcessAudioCapture.dll）は x86_64 / arm64 の両方を用意してから当該アーキを「対応」と宣言する
5. **実行時にネットワークへ出ない**: VAD モデル等の全アセットを同梱し、ダウンロード・オンライン認証・テレメトリを一切行わない。エアギャップ PC（USB 配布・ネット未接続）での完全動作を保証する

## 2. スコープと非スコープ

### 2.1 スコープ（pyflexaudio が責任を持つ範囲）

- **音声ソース キャプチャ** × 9 マス（Win/Mac/Linux × Mic/System/Process）
- **Source → Processor → Sink パイプライン** + fan-out 配信
- **EventBus**（Level / SpeechStart/End / Error / DeviceDisconnected / StateChanged ほか）
- **基本 Processor**: LevelMeter / Resample / ChannelConvert
- **基本 Sink**: CallbackSink / FileSink (WAV)
- **VAD アドオン**: `pyflexaudio_vad` サブパッケージとして同梱（依存は分離）
- **CLI**: 動作確認用の最小 CLI

### 2.2 非スコープ（やらない）

- **ASR・話者分離**（whisper.cpp / sherpa-onnx の領域）
- **要約・翻訳**（LLM の領域）
- **GUI**（WhisperApp の領域）
- **録音セッション管理ロジック**（複数ファイル管理・メタデータ等は呼び出し側）
- **クラウド配信**
- **モバイル**（Android / iOS は別ライブラリで対応）

### 2.3 グレーゾーン（議論余地あり）

- ホットスワップ（実行中 Source 切替）→ 7.3 で議論
- crash-safe WAV の自動分割 → 8.1 で議論

## 4. 依存方針（採用 vs 自前 判断マトリクス）

設計哲学に基づき、現行 v0.1.0 の全依存を再評価する。

### 4.1 確定方針

| 依存 | 現状 | 再設計後 | 判断理由 |
|------|------|---------|----------|
| **numpy** | コア依存 | ✅ 継続 | 置換不可、誰でも使う、配列計算の事実上の標準 |
| **sounddevice** (PortAudio) | コア依存 | ✅ 継続（optional 化） | PortAudio 抽象は実用的。自前で Win/Mac/Linux の Mic を書く工数（推定 1,500 行）に見合わない |
| **miniaudio** | フォールバック | ❌ 削除 | sounddevice で十分。フォールバック維持コスト > 価値 |
| **soxr** | コア依存 | △ optional 化 | リサンプル必要時のみ。自前 polyphase 実装は品質維持が困難なので採用継続だが optional |
| **onnxruntime** | optional (VAD) | ✅ 継続（vad アドオン側のみ） | Silero VAD ONNX 推論に必須。コア依存から完全分離 |
| **pyobjc-framework-\*** | optional (Mac) | ✅ 継続 | SCK / CoreAudio を Python から触る唯一の現実的手段 |
| **pyaudiowpatch** | optional (Win system) | 🔍 **要再評価** | WASAPI Loopback の取得だけなら ctypes で 200〜300 行で書ける。「PortAudio 経由の Loopback」は 3 段ラッパー。Phase 1 で自前 PoC 検討 |
| **pyflexaudio-win-dll** (ProcessAudioCapture) | optional (Win process) | ✅ 継続 | 自前実装の極致。継続利用 |
| **pulsectl** | optional (Linux) | 🔍 **§ 6 で実機検証** | PulseAudio C API ラッパー。「PulseAudio 経由 PipeWire」も視野。自前 ctypes と比較検討 |
| **pipewire-python** (新規候補) | — | 🔍 **§ 6 で実機検証** | PipeWire のラッパー。メンテ活発度と機能網羅性を要評価 |

### 4.2 削除候補のサマリ

| 削除候補 | 理由 | 削減見込み |
|---------|------|-----------|
| miniaudio (mic フォールバック) | sounddevice で代替可、フォールバック価値低 | コード -110 行、依存 -数MB |
| 既存 `registry.py` | Source 動的登録は YAGNI | コード -69 行、テスト -144 行 |
| 既存 `permissions.py` | OS 権限ダイアログ抽象は薄い、各 OS で別実装が結局必要 | コード -96 行 |
| 既存 `processors/vad.py` をコアから | サブパッケージへ移動 | コア -183 行、テスト -1,021 行 |
| 既存テストの edge_cases / additional_paths | LLM 自動生成肥大化、本質的価値要再評価 | テスト -900 行前後 |

**削減合計見込み**: コア実装 -460 行 / コアテスト -2,000 行強 / 依存 onnxruntime 完全分離

### 4.3 採用継続の正当化メモ

- **sounddevice 継続の理由**: 自前 WASAPI + CoreAudio + ALSA 実装は最低 2,000 行クラス。「ProcessAudioCapture DLL は WASAPI を自前で書いた」前例があるものの、3 OS 分のマイク入力までスコープを広げると工数対効果が悪い。sounddevice は単一段の薄いラッパー（PortAudio 直結）であり 4.1 の判断基準を満たす
- **soxr 継続の理由**: 高品質リサンプリングは VAD/ASR 用途で品質劣化が許されない。numpy で書ける scipy ライクな実装ではエイリアシング対策が不十分

## 5. 9 マスマトリクスの実装方針

| | Windows | macOS | Linux |
|---|---|---|---|
| **マイク** | sounddevice 継続 | sounddevice 継続 | sounddevice 継続 |
| **システム音声** | WASAPI Loopback（pyaudiowpatch or 自前 ctypes） | ScreenCaptureKit | **§ 6 で検証** |
| **プロセス別** | ProcessAudioCapture DLL 継続 | ScreenCaptureKit | **§ 6 で検証** |

### 5.1 Win × Mic — sounddevice 継続

実装方針: 現行 `mic_sounddevice.py` をベースに、`indata.copy()` + `put_nowait` + `DROP_OLDEST` の鉄則を継続。コールバックスレッド内でのブロック禁止。

### 5.2 Win × System — pyaudiowpatch 継続 or 自前 PoC

**現状**: pyaudiowpatch は PortAudio + WASAPI Loopback パッチ版。3 段ラッパー（pyaudiowpatch → PyAudio → PortAudio → WASAPI）。

**判断プロセス**:
- Phase 1 着手時に **自前 WASAPI Loopback の PoC を 200〜300 行で書く**
- 比較項目: レイテンシ / CPU 負荷 / コード行数 / 依存サイズ / 保守性
- 自前実装の参考: ProcessAudioCapture DLL の WASAPI 周りの C++ コード

### 5.3 Win × Process — ProcessAudioCapture DLL 継続

実装方針: 現行 `process_windows.py` のままで本質的に問題なし。改善点：
- ポーリングループの 10ms sleep を見直し（イベントドリブン化検討）
- `pyflexaudio-win-dll` のバージョン上限固定
- DLL 関数シグネチャ宣言を別モジュールに分離
- **`g_lastError` のハンドル単位化**（DLL 側修正）: 現状の DLL は `static wchar_t g_lastError[512]` が全ハンドル共有のグローバルバッファ。複数キャプチャが同時にエラーを出すと最後の書き込みが前を上書きし、どのハンドルのエラーか判別不能。マルチキャプチャ前提なら `PacGetLastError(handle)` のハンドル単位化、またはエラーコードのみ返す設計へ変更（詳細は § 12.1）
- **ARM64 ビルドの追加**: 現状 `prebuilt/x64/` のみ。Windows on ARM 対応には ARM64 ビルドが必須（詳細は § 12.2）

### 5.4 Mac × Mic — sounddevice 継続

特筆事項なし。

### 5.5 Mac × System — ScreenCaptureKit 継続

実装方針: 現行 `system_macos.py` の `_SCKBaseBackend` 継続。改善点：
- pyobjc の遅延 import を維持
- planar → interleaved 変換のコードを別関数に切り出し
- フォーマット検出ロジックを単純化

### 5.6 Mac × Process — ScreenCaptureKit 継続

実装方針: 現行 `process_macos.py` で `_build_content_filter()` のみ差し替える Template Method を維持。これは v0.1.0 の優れた設計判断。

### 5.7 Linux × Mic — sounddevice 継続

特筆事項: sounddevice は ALSA/PulseAudio/PipeWire（pipewire-pulse 経由）すべてで動作する。

### 5.8 Linux × System — § 6 検証で確定

候補:
- **PulseAudio monitor source** via pulsectl
- **PulseAudio monitor source** via libpulse ctypes 自前
- **PipeWire native** via pipewire-python
- **PipeWire native** via libpipewire ctypes 自前

WASAPI Loopback の Linux 版に相当するのが「monitor source」。スピーカー出力ごとに自動生成される録音用仮想入力。

### 5.9 Linux × Process — § 6 検証で確定

候補:
- **PipeWire node graph + link API** — 対象アプリの出力ノードを特定し録音ノードへ link
- **PulseAudio module-loopback** ハック — sink-input を loopback で録音
- **WirePlumber filter-chain** — ルーティング設定経由

Windows ProcessAudioCapture と Mac SCK のような統合 API は Linux に存在しない。**全候補が「ハック寄り」**である点を計画書として明記する。

## 6. Linux 実機検証計画

「とりあえず実機でテスト、既存ライブラリより自前を考える」方針に基づく検証プロセス。

### 6.1 検証環境

- **メイン検証マシン**: Ubuntu ラップトップ（ユーザー所有）
- **検証 OS バージョン**: Ubuntu 確定後に記録（PipeWire/PulseAudio 採用状況に直結）
- **補助**: homelab Linux サーバー（GUI なし、PulseAudio コマンド検証用）

### 6.2 検証ステップ

**Step 1: 環境調査（30 分）**

pactl info | grep "Server Name"   # "PulseAudio" or "PulseAudio (on PipeWire)"

- `parecord` でシステム音声録音 → WAV 化して再生確認
- `pw-record` で同じことを PipeWire native で試す
- 「最低限の発射台」が動くことを確認

**Step 3: ライブラリ候補の最小実装比較（半日 × 候補数）**

各候補について、**最小コード（システム音声を 5 秒キャプチャして WAV 書き出し）** を書く:

| 候補 | 最小実装目安 | 依存サイズ | メンテ活発度 |
|------|------------|----------|------------|
| pulsectl | ~30 行 | 数百KB | 要確認 |
| libpulse via ctypes 自前 | ~150 行 | 0（libpulse はシステム提供） | — |
| pipewire-python | ~50 行 | 数MB | 要確認 |
| libpipewire via ctypes 自前 | ~300 行 | 0（libpipewire はシステム提供） | — |

**Step 4: プロセス別音声の PoC（1〜2 日）**

最も難しい部分。各方式の難易度を実機で確かめる:
- 対象アプリ: 例えば `mpv` で動画再生 → その音声だけ録音
- PipeWire node graph 方式: `pw-link` でルーティングを試す → Python で再現
- PulseAudio module-loopback 方式: `pactl load-module module-loopback ...` → Python で再現

**Step 5: 計測**

各候補で以下を測定:

| 指標 | 計測方法 | 合格ライン |
|------|---------|----------|
| キャプチャレイテンシ | ループバック信号で測定 | < 100 ms |
| CPU 負荷（idle） | `top` / `pidstat` | < 5% |
| メモリフットプリント | `pmap` | < 30 MB |
| バンドルサイズ影響 | `pip show` の Requires | 最小化 |
| 実装行数見積もり | 9 マス全体での総量 | < 500 行 |

**Step 6: 採用判断（半日）**

### 6.3 想定される判断パターン

- **パターン A**: PipeWire が圧倒的に主流（Ubuntu 22.10+ 利用者多数） → PipeWire native で統一
- **パターン B**: PulseAudio 互換層で十分 → pulsectl 採用または自前 ctypes（行数次第）
- **パターン C**: 両方サポートが必要 → ランタイムで検出し切替

「無駄な接続点削減」の原則からは **A または B の単一方式** が望ましい。C は最後の選択肢。

## 7. API 設計（完全刷新）

WhisperApp 未統合 = 破壊的変更 OK。v0.1.0 の API は参考程度に、ゼロから設計する。

### 7.1 v0.1.0 API の問題点

- `FlexAudioSession` が **責務過多**（状態機械 + Source 管理 + Sink 管理 + Pipeline 制御 + イベント発行）→ 355 行
- `LevelMeterProcessor` が **常時 ON** で OFF 不可（オーバーヘッド固定）
- `set_source()` の引数 `device_index | pid | mode` が **型的にユニオン的**で曖昧
- イベントの型が多すぎる（10 種以上）

### 7.2 刷新案（ドラフト、§ 別途詳細化）

```python
# === Capture: 1 つの Source からの取り込み ===
with pyflexaudio.capture_microphone(device=None) as cap:
    for chunk in cap.chunks():   # generator API
        do_something(chunk)

# === Session: 複数 Sink への fan-out + ライフサイクル管理 ===
session = pyflexaudio.Session()
session.add_source(pyflexaudio.SystemAudio())
session.add_sink(pyflexaudio.FileSink("out.wav"))
session.add_sink(pyflexaudio.CallbackSink(my_handler))

with session:
    session.start()
    # ...
    session.stop()

# === VAD アドオン ===
from pyflexaudio_vad import SileroVAD
session.add_processor(SileroVAD(threshold=0.5))
session.on("speech_start", lambda ev: print("発話開始"))
```

**変更ポイント**:
- 生 generator API（小規模用途）と Session API（複合用途）の **二層化**
- `LevelMeter` をデフォルトで切る（必要な時だけ `add_processor`）
- `set_source` をなくし `add_source` + `remove_source`（ホットスワップは別 API）
- イベント名を文字列にしてユーザー定義拡張可能に（型は内部で検査）

### 7.3 ホットスワップの扱い

v0.1.0 の `set_source` 中切替は実装複雑度が高い割に用途が限定的。**Phase 1 では非対応**にして、Session を捨てて作り直す（stop → 新 Session）方針。需要があれば後フェーズで追加。

### 7.4 詳細は別文書

詳細 API 設計は `docs/API設計書.md` に分離して書く。本書では方向性のみ確定。

3. **VAD テスト分離**: `tests/vad/` に隔離（onnxruntime 必要）
4. **ハードウェア依存テスト**: `@pytest.mark.hardware` で分離し CI 不要化（v0.1.0 で既に実施済み）

|---------|------|-------|
| smoke | 各 OS で各 Source が起動するか | ★★★ |

## 12. 横断リスク（同時キャプチャ・アーキ差）

OS や 9 マスを横断して効いてくるリスク。各マスの実装方針（§ 5）とは独立に、全体として担保すべき事項。

### 12.1 同時キャプチャの競合

「同じデバイス／同じプロセスを複数が録音する」を 2 つに分けて扱う。

#### (A) OS/API レベルの同時録音可否

| API | 同一デバイス/PID の多重キャプチャ | 競合リスク |
|-----|------------------------------|-----------|
| WASAPI 共有モード（マイク・Loopback） | ✅ OS がミックス/コピーして配る | 低。ただし**排他モード**を誰かが掴むと締め出される |
| WASAPI Process Loopback（自前 DLL） | ✅ PID ごとに独立クライアント | 低〜中（DLL のグローバル状態に注意、後述） |
| macOS ScreenCaptureKit | ✅ 各 SCStream が独立 | 中。ストリーム数増でリソース/権限負荷 |
| PipeWire/PulseAudio monitor | ✅ サーバーが fan-out | 低。Linux は多重読み取り前提 |

**原則**: 現代 OS のオーディオは「共有モードなら多重キャプチャは基本 OK」。OS 側がミキサー/コピーを担う。唯一の地雷は**排他モード**（§ 1.4 不変条件 2 で禁止）。

#### (B) ProcessAudioCapture DLL の実態（2026-06-03 ソース検証済み）

| 観点 | 実態 | 評価 |
|------|------|------|
| 異なる PID を同時キャプチャ | `PacStartCaptureEx` ごとに `new CLoopbackCapture`（独立オブジェクト） | ✅ 安全 |
| 同一 PID を 2 ハンドルで同時キャプチャ | WASAPI Process Loopback は PID ごと独立クライアント生成可 | ✅ OS 的には可能（**要実機確認**） |
| エラー報告 | `static wchar_t g_lastError[512]` が**全ハンドル共有のグローバルバッファ** | ⚠️ **競合**。2 キャプチャ同時エラーで last-writer-wins、ハンドル判別不能 → § 5.3 で修正 |
| 「Already capturing」 | `m_lastError`（インスタンスメンバ） | ✅ ハンドル単位で OK |

#### (C) pyflexaudio 自身が抱える「内側の競合」（最重要）

OS が許しても、ライブラリ設計で誤動作を生むパターン。

| リスク | 説明 | 対策 |
|--------|------|------|
| フィードバックループ | システム音声キャプチャ中に再生で無限ループ | 全 OS で自プロセス除外（§ 1.4 不変条件 3） |
| フォーマット不一致 | 別サンプルレートの 2 Source を同一 Sink に fan-out → WAV 破損 | Sink 側でフォーマット固定 or リサンプル必須化 |
| デバイス切断イベントの多重発火 | 同一デバイスを 2 Session が開く → 二重 stop | イベントのハンドル紐付け |
| リングバッファ取りこぼし | 複数キャプチャで polling が CPU 競合 → overrun | イベントドリブン化 or polling 間隔調整 |

### 12.2 CPU アーキテクチャ差（ARM / x86）

**結論: 「データは安全、バイナリ依存が危険」**

#### (A) 層ごとの安全性

| 層 | ARM↔x86 で問題か | 理由 |
|----|----------------|------|
| PCM データのバイト順 | ✅ 問題なし | ARM/x86 とも実運用はリトルエンディアン。`np.frombuffer` そのまま動作 |
| float32 表現 | ✅ 問題なし | 両者 IEEE 754 |
| numpy 演算結果 | ✅ ほぼ問題なし | SIMD（AVX/NEON）実装差はあるが結果一致 |
| **ネイティブ依存のビルド** | 🔴 **本丸** | アーキごと別バイナリが必要 |

#### (B) アーキ別 依存可用性マトリクス

| 依存 | x86_64 | ARM64 | 危険度 |
|------|--------|-------|-------|
| **ProcessAudioCapture.dll** | ✅ `prebuilt/x64/` のみ | 🔴 **ビルドなし** | **高**。Win on ARM には ARM64 再ビルド必須 |
| onnxruntime（VAD） | ✅ | △ ARM 版あるが Win-ARM は限定的 | 中。Mac Apple Silicon は OK |
| numpy / soxr | ✅ | ✅ aarch64 wheel | 低 |
| sounddevice/PortAudio | ✅ | ✅ | 低 |
| pyobjc（Mac SCK） | ✅ | ✅ universal2 | 低（Apple Silicon ネイティブ） |
| PipeWire/PulseAudio（Linux） | ✅ | ✅ システム提供 | 低 |

#### (C) アーキ × OS の実機検証マトリクス

| 環境 | 優先度 | 確認事項 |
|------|-------|---------|
| Apple Silicon Mac (arm64) | ★★★ 既定 | SCK / VAD 推論速度 |
| x86_64 Win | ★★★ 既定 | 現行ターゲット |
| Windows on ARM (arm64) | ★★☆ | DLL の ARM64 ビルド可否、onnxruntime-ARM |
| x86_64 Linux (Ubuntu ラップトップ) | ★★★ | Linux 検証の主戦場 |
| ARM64 Linux (RasPi / ARM サーバー) | ★☆☆ | 余裕があれば。aarch64 wheel 可用性 |

**性能差の注意**: onnxruntime の VAD 推論は ARM（NEON）で x86（AVX2）より遅いことがある。リアルタイム発話検出が間に合うか実機計測が必要。弱い ARM（RasPi 級）では VAD がボトルネック化しうる。

### 12.3 横断リスクの検証タスク（§ 6 検証計画に統合）

- **同時キャプチャ test**: 2 プロセスで同一デバイス/同一 PID を録音 → 両方正常に音が録れるか、片方が締め出されないか
- **フィードバックループ test**: システム音声キャプチャ中に音声再生 → 自プロセス除外が効くか
- **ARM64 ビルド test**: ProcessAudioCapture.dll を ARM64 ツールチェーンでビルド → x86 固有命令やライブラリ依存がないか
- **VAD 推論速度 test**: Apple Silicon / ARM Linux で VAD のリアルタイム性を計測
